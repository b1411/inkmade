## 🔴 CRITICAL — блокируют продакшен

### C1. Реальный SUPABASE_SERVICE_ROLE_KEY лежит в `.env`
`.env:6` — реальный `sb_secret_...` ключ, обходит весь RLS. Ротировать в Supabase Dashboard, держать только в Vercel env.

### C2. `mock-confirm` доступен в проде без guard
`server/api/payment/mock-confirm.post.ts` — любой залогиненный юзер вызывает → бесплатная «оплата» своего заказа. (нашли 2 агента независимо). Добавить `if (NODE_ENV==='production') throw 404` или исключить роут из прод-сборки.

### C3. Три артефакта дизайна (§13.2) НЕ реализованы — принт нельзя распечатать
`useDesign.ts:133-154`, `customize/[id].vue:64` — `asset_url` = `URL.createObjectURL(file)` (blob-URL, живёт только во вкладке). `Konva.toBlob()` нигде не вызывается, оригинал в Storage не грузится. После закрытия вкладки оператор не получит файл. Нужно: загрузка оригинала + превью в Storage + скриншот композиции перед `cart.add()`.

### C4. `px_per_mm` — глобальная константа 1.1 вместо per-mockup `anchor` (§10, §11.4)
`useDesign.ts:8` — спецификация в мм для цеха систематически искажена (разные мокапы → разный масштаб). (нашли 2 агента). Нужна колонка `print_zones.anchor jsonb` / расчёт из ширины мокапа, использовать в `toMm()`/`zoneRect()`.

### C5. `user_consents` отсутствует полностью — юр.риск РК
0 совпадений в миграциях и формах. Нет таблицы, нет чекбокса ToS при register/checkout. Закон РК о ПД: невозможно доказать согласие. Нужно: миграция `user_consents` + чекбокс + запись факта.

### C6. Нет транзакционности: status+log и applyPaid пишутся раздельными запросами
`server/api/orders/[id]/status.post.ts:45-50`, `server/utils/payment.ts:48-78` — сбой между await оставляет данные в частичном состоянии; лог перестаёт быть источником истины. Двойной webhook при гонке → двойное списание склада (идемпотентность только по `paid_at`). Вынести в Postgres-функцию (`svc.rpc`) с транзакцией + advisory lock.

### C7. Цена берётся из localStorage, сервер не перепроверяет
`useOrder.ts:27,36-43` — подмена `unitPrice=1` в localStorage → заказ на 1 тенге. Серверный пересчёт цены по `variant_id` из БД при создании заказа.

## 🟠 HIGH

- **H1.** `PAYMENT_WEBHOOK_SECRET=dev-mock-secret` (публичный дефолт) реально в `.env` + fallback в коде `webhook.post.ts:13`. Убрать fallback, сгенерировать случайный секрет.

- **H3.** `order_items` INSERT-политика без `and o.status in ('created','pending')` (`0004_rls.sql:146`) — добавление позиций в оплаченный заказ.
