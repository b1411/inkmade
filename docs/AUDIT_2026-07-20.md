## 2. Сквозные темы

Находки, подтверждённые независимо несколькими агентами — самые надёжные.

### 2.1. Репозиторий перестал быть источником истины о проде

Три агента пришли к этому разными путями.

- **Миграции 0091, 0092 не применены.** Прод на `20260717050702 / 0090_epay_payment_foundation`.
  В проде нет ни таблицы `shop_payouts`, ни `request_shop_payout`, ни `process_shop_payout`,
  ни колонки `status_changed_at` в сигнатуре `studio_list_queue`.
- **Код для них уже в `main` и задеплоен**, потому что `FEATURES.b2bStorefront = true`
  ([features.ts:27](../shared/config/features.ts#L27)).
- **`app/types/database.types.ts` врёт** — типы сгенерированы из локальных миграций,
  поэтому typecheck зелёный, а ломается только рантайм.
- **3 объекта в проде без локального файла:** `studio_rpc_revoke_anon`,
  `0079b_revoke_public_admin_rpc`, `print_zone_canvas_bounds_check_fix`.
  → `supabase db reset` локально даёт **другую схему** с другими грантами и другим CHECK.
- **В CI/CD нет ни одного шага применения миграций.** `supabase db push` не вызывается нигде.

### 2.2. Ни один гейт качества не защищает `main`

[.github/workflows/ci.yml:29,71,99](../.github/workflows/ci.yml#L29) вызывает `npm ci`,
но в репо трекается только `pnpm-lock.yaml`, а `package-lock.json` — в
[.gitignore:37](../.gitignore#L37).

Проверено прямым прогоном:

```
npm error code EUSAGE
npm error The `npm ci` command can only install with an existing package-lock.json
```

`actions/setup-node` с `cache: npm` ([ci.yml:26](../.github/workflows/ci.yml#L26)) падает
ещё раньше. Так продолжается с коммита `e440342` («use pnpm only», 2026-06-28) — то есть
**~3 недели** typecheck, lint, 231 тест и build не гейтили ни один PR. Именно поэтому
несобирающийся коммит доехал до `main` и был запушен в `origin`.

Смежное: [vercel.json:3](../vercel.json#L3) — `"installCommand": "npm install"` при
закоммиченном `pnpm-lock.yaml`. Все 18 зависимостей объявлены через `^`. Каждый деплой
резолвит свежие миноры → **прод собирается из набора пакетов, который никто не тестировал**,
и откат на прошлый коммит не воспроизводит прошлую сборку. Плюс `overrides` из
`pnpm-workspace.yaml` npm не читает вовсе.

### 2.3. Деньги: логика сильная, контур разомкнут

SQL-инварианты — лучшая часть проекта (advisory-локи, сверка сумм, TOCTOU-гарды,
guard-триггеры, реверс роялти). На уровне БД это ~85%. Но:

- Платёжный шлюз **ни разу не исполнялся против банка** — в проде 4 записи `payments`,
  все `provider='mock'` от 2026-06-07, ни одной `epay`.
- Фискализации РК (онлайн-ККМ/ОФД) **нет как кода** — только env-флаг в readiness.
- Ни одного таймаута на четырёх вызовах к ePay
  ([payment.ts:140,165,191,217](../server/utils/payment.ts#L140)).
- Нет крона сверки зависших `pending` — cron/edge-функций у проекта нет вообще.
- `acquiring_fee` и `shipping` **никогда не пишутся** в `finance_entries` → прибыль на всех
  дашбордах завышена.

### 2.4. Печатный файл не гарантирован

Ради этого артефакта построен весь продукт, и он — самое слабое место.

- **11 из 16 прод-зон имеют `bounds_canvas = NULL`**, включая `tshirt/chest`, `tshirt/back`,
  оба рукава, `polo`, `sweatshirt`, `cap`. Работает фолбэк
  ([useDesign.ts:201-209](../app/composables/useDesign.ts#L201)), который в собственном
  комментарии описан как «рисует зону НЕ ТАМ и втрое крупнее».
- `print_files` в схеме — `optional()`
  ([schemas.ts:53](../server/utils/schemas.ts#L53)). Единственный гард — клиентский.
- **В проде нет ни одного дизайна с печатным файлом** — все 4 `order_items`, ссылающиеся
  на `designs`, указывают на записи с `print_file_url IS NULL`.

### 2.5. Rate limiting по конструкции не работает

`Map` в памяти инстанса ([rate-limit.ts:26](../server/middleware/rate-limit.ts#L26)) на
эфемерных лямбдах Vercel + гейт `path.startsWith('/api/')`, который не покрывает ничего
из того, что клиент дёргает в Supabase напрямую: `auth/v1/token` (перебор пароля),
`rpc/shop_storefront` (перебор кода закрытого магазина), `rpc/shop_track`, `storage/v1/object`.
Капчи/бот-защиты нет нигде (0 совпадений по `captcha|turnstile|hcaptcha|recaptcha`).

### C4. `shop_earnings.status = 'reversed'` против CHECK-констрейнта
Подтверждено **двумя агентами независимо**.

Прод-констрейнт: `shop_earnings_status_check CHECK (status = ANY (ARRAY['accrued','paid']))`.
Обе функции из `0084` — `refund_order` и `change_order_status` (`paid→cancelled`) — выполняют
`update shop_earnings set status='reversed'`. У `royalty_earnings` `'reversed'` в CHECK есть,
у `shop_earnings` — нет.

**Первый же возврат или отмена оплаченного B2B-заказа падает с `23514 check_violation`
и откатывает всю транзакцию.** Деньги не возвращены, статус не изменён, админ видит сырую
ошибку Postgres. Не выстрелило только потому, что `shop_earnings` пуста.

**Done:** миграция, расширяющая CHECK до `('accrued','paid','reversed')` + регресс-тест.

### C5. Миграции 0091/0092 не применены при живом UI
См. §2.1. Владелец магазина жмёт «Запросить выплату» → PostgREST 404. Вкладка выплат
в админке падает. SLA-плашки очереди студии считаются от `undefined`.

### C6. Кнопка «Оплачен» в админ-карточке ломает заказ
[admin/orders/[id].vue:50](../app/pages/admin/orders/[id].vue#L50) →
[status.post.ts:31](../server/api/orders/[id]/status.post.ts#L31) → `change_order_status`
делает голый `update orders set status='paid'` **мимо `apply_paid`**: не ставится `paid_at`,
не списывается сток, не пишутся `revenue`/`cogs`/роялти.

Заказ становится **невидим для цеха** — `studio_list_queue` фильтрует по `paid_at is not null`.
Одновременно возврат по нему невозможен («Заказ не был оплачен»).

### C7. Платёжный шлюз никогда не работал + нет фискализации
См. §2.3. Два конкретных подозрения в коде: `epayToken()` запрашивает токен без
`invoiceID/amount/currency/terminal`; `refund()` шлёт `amount` в теле, а не query-параметром.

### C8. Печатный артефакт необязателен и может уйти неполным
См. §2.4. Плюс: при мультизонном экспорте зона молча пропускается при `toBlob → null`
(типично для iOS Safari), а гард проверяет только `files.length` — дизайн «грудь + спина»
уедет в производство с одним файлом из двух.

## 4. HIGH — по доменам

### Деньги
- Возврат **не идемпотентен**: `refund.post.ts:22-34` читает статус до вызова банка, без лока
  → двойной клик = двойной возврат денег.
- Ответ банка на возврат **нигде не сохраняется** — при сбое `refund_order` деньги ушли,
  следов нет.
- `/api/payment/create` не проверяет статус заказа → **отменённый заказ можно оплатить**.
- **Повторная оплата невозможна**: `payment_invoice_id` уникален на заказ, второй `POST /invoice`
  с тем же id банк отклонит → заказ невозможно оплатить никогда.
- Нет реконсиляции зависших `pending`.
- `acquiring_fee` / `shipping` не попадают в P&L.

### Безопасность
- Бакет `design-uploads` — **публичный + анонимная запись без ограничений**
  (`0008:24-29`). Хардеринг, обещанный в комментарии самой миграции, не сделан за 84 миграции.
- **Печатные файлы и исходники клиентов лежат в этом публичном бакете.** Защита —
  только неугадываемость имени. Приватный бакет `designs` с правильными политиками
  существует, но не используется.

### Конструктор
- Печатный файл молча даунскейлится (`PRINT_CAP_PX = 4096`), но в метаданных остаётся
  `dpi: 300`. Зона «Спина» 300×400 мм → фактически **260 DPI под видом 300**. Цех печатает
  по цифре из спецификации.
- PDF: **белая подложка вместо прозрачности** + фиктивный бейдж «вектор». Логотип с
  прозрачным фоном → белая плашка на чёрной футболке.
- Удаление фона **не работает в проде** (двойной блокер): `import` по переменной с
  `@vite-ignore` не бандлится Vite + CSP без `wasm-unsafe-eval` и без хоста моделей.
- После удаления фона холст и печатный файл используют **старую картинку** (кэш образов
  ключуется по `p.id`, а не по URL).
- **Шелкография продаётся по числу цветов, но артефакт под неё не готовится** — экспорт
  всегда отдаёт плоский RGBA-PNG, ограничений на градиенты/фото нет.

### Бэк-офис
- Персонал **не получает ни одного уведомления** о новом заказе. Единственный сигнал —
  Realtime на открытой вкладке.
- Одобрение клиентской заявки на возврат **не делает ничего, кроме тоста**.
- Ручной заказ нельзя отгрузить — нет ни адреса, ни телефона, ни пути к оплате.
- Клиента не уведомляют об `on_hold` / `reprint` / `cancelled` / `refunded`.
- Списки грузятся целиком без пагинации → **сводные KPI врут** после ~1000 строк.

### B2B
- Дублирующийся `shopItemId` в корзине ломает распределение скидки — **недостачу платит
  платформа**. Это самый типичный B2B-сценарий (несколько размеров одной позиции).
- Удаление дизайна тихо ломает опубликованную позицию витрины (FK `SET NULL`, `is_active`
  остаётся `true`).
- **Субдомены: резолвера нет.** «Включение одним флагом» — миф. Нет рендера витрины на `/`
  тенант-хоста, нет wildcard-DNS/rewrite, `provisionSubdomain` — заглушка `not_implemented`.

### Схема БД
- Удаление товара каскадом **обнуляет историю заказов** — `order_items` не хранит снапшот
  (`product_title`, `sku`, `size`, `color_name`).
- **Ни одного CHECK на неотрицательность денег** — `orders.total` может стать отрицательным.
- `refund_order` не восстанавливает склад и не оставляет платёжного следа.
- `shop_earnings.order_id ON DELETE CASCADE` уничтожает финансовый след, а `shop_balances`
  остаётся завышенным → владелец выведет деньги за несуществующий заказ.

### Инфра
- Нет `maxDuration` и `regions`. Функции по умолчанию в `iad1` (US East), а Supabase —
  в `eu-central-1`, пользователи в Казахстане. AI-генерация: 90 с × 2 попытки — не влезет
  в дефолтный лимит.
- **Алертинга и uptime-мониторинга нет.** `error_monitoring` помечен `critical: false` —
  прод-гейт пропускает запуск вообще без канала ошибок.

### Фронтенд
- **Казахская версия невидима для поиска** — `no_prefix` без `hreflang` и `baseUrl`,
  локаль только в cookie. Половина заявленного рынка.
- **Клавиатурой нельзя выбрать слой в конструкторе** — `<li>` с `@click` без `tabindex`
  и `role`, при этом кнопки удаления в таб-порядке есть.
- ~120 строк UI живут в теневых словарях внутри `.vue`; 7 landing-компонентов не содержат
  ни одного `t()`.
- Лендинг рендерится синхронно поверх 4 запросов в Supabase — ни ISR, ни SWR, ни prerender.

### Тесты
- **Ноль тестов на серверном денежном контуре.** Ни один из 24 файлов `server/api/**`
  не импортируется ни одним тестом.
- Харнесс это исключает: `environment: 'node'`, `@nuxt/test-utils` и DOM в `package.json`
  отсутствуют. 33 composable, ~90 компонентов и 24 nitro-маршрута **невозможно** покрыть.
- `tests/shop-payout.test.ts` и `tests/studio-queue-sla.test.ts` — это `expect(sql).toContain(...)`
  по тексту миграции. **Ложный зелёный ровно на выплатах.**
- E2E не пройдут в CI и после починки: конфиг объявляет 5 проектов, CI ставит только chromium;
  визуальные бейзлайны закоммичены как `*-win32.png`, а CI — ubuntu.

- **Паритет ключей ru↔kk — 100%** по всем 16 namespace.
- **Качество кода высокое**: один TODO на весь проект, ноль `@ts-ignore`, ноль забытых
