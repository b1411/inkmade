# INKMADE — Аудит ролей и детальный план развития

Документ дополняет `INKMADE_roles_crm.md`. Здесь: результат сквозного аудита реальной реализации каждой роли (что работает / что заглушка / чего нет), сводка критичных багов и **детальный план по каждой роли — что доделываем + какие крутые фишки вводим**.

Дата аудита: 2026-06-07. Метод: чтение всех страниц/composables/миграций по 4 ролям + сверка со спецификацией.

---

## 0. Сводка готовности (TL;DR)

| Роль | Готовность | Состояние | Главный риск |
|---|---|---|---|
| **customer** | ~70% | Ядро покупки работает целиком | Нет чеков, нет email-уведомлений, теряются гостевые дизайны |
| **designer** | ~82% | Инфраструктура роялти боевая | Нет reversal роялти при возврате, нет публичной витрины |
| **operator** | ~90% | Производство почти готово | 🚨 **Утечка цен/себестоимости оператору** |
| **admin** | ~65% | CRM-каркас есть | 🚨 **Чистая прибыль считается неверно** (= выручке) |

### 🚨 Блокеры до продакшена (исправить первыми)
1. **[operator] Утечка финансов.** `useStudio.ts` тянет `order_items(*)` и `total` → оператор видит `unit_price`, `unit_cost`, выручку. Прямое нарушение §5.2. → выбирать только производственные поля + убрать `total` из доски.
2. **[admin] Неверная прибыль.** RPC `admin_finance_stats` (`0014_admin_stats.sql`) возвращает `profit = revenue`, не вычитая cogs/роялти/эквайринг/доставку. Все финрешения на ложных числах.
3. **[designer] Нет reversal роялти.** При возврате/отмене оплаченного заказа `royalty_earnings` не уходит в `reversed` — дизайнер получает деньги за отменённый заказ. §7.3 нарушен.
4. **[customer] Нет фискального чека** в UI заказа (`fiscal_receipt_url` есть в БД, не выводится). Для B2C в KZ обязательно.
5. **[customer] Нет email-уведомлений** о смене статуса (webhook есть, писем нет) — бьёт по retention.

## 1. CUSTOMER (`/account`) — покупатель

### 1.1 Что РАБОТАЕТ ✅
- Каталог → товар → кастомайзер (Konva) → корзина → checkout → mock-оплата — полный флоу.
- Мои заказы: список + укрупнённый статус (5 этапов) + карточка с позициями/превью/треком.
- Realtime-трекинг статуса на `/order/[id]`.
- Повтор заказа в один клик (на карточке заказа).
- Мои дизайны: галерея, «поделиться по ссылке» (share_token → `/design/[token]`).
- Избранное: товары + принты, toggle с карточки товара.
- Адреса: создать/удалить/сделать дефолтным + автозаполнение checkout.
- Профиль: имя, телефон, email (read-only), смена пароля.
- Гостевой флоу: сборка корзины без логина → логин на checkout → редирект назад.

### 1.2 Что ЧАСТИЧНО 🟡 / НЕТ ❌

| Подарочные заказы | ❌ | Нет `is_gift`, формы получателя, открытки |
| Перенос гостевых **дизайнов** в аккаунт | ❌ | Корзина переносится, дизайн (холст/placements) — нет |

### 1.3 ПЛАН — доделать (приоритет)
**P0 (до запуска):**
- Фискальный чек: endpoint + кнопка «Скачать чек» на `/order/[id]`.
- Email-уведомления на каждый переход статуса (paid / в производстве / отгружен / доставлен).
- Перенос гостевых дизайнов: при логине на checkout сохранять локальные designs как черновики (`is_saved=false`).

**P1:**
- Редактирование адреса (форма edit).
- Кнопка «Заказать снова» прямо в карточке дизайна и в строке списка заказов.
- Версионирование дизайна (`parent_design_id`): «обновить» vs «сохранить как новую версию».
- Валидация телефона через `libphonenumber-js`.

### 1.4 🔥 Крутые фишки для customer
1. **Подарочный флоу с цифровой открыткой.** Чекбокс «это подарок» на checkout → имя получателя + текст открытки + дата вручления. Скрытие цены в упаковке. Генерация красивой e-открытки со ссылкой на трекинг для получателя.
2. **«Примерь на себе» (AR/preview-mockup).** Наложение принта на фото-мокап изделия выбранного цвета прямо в кастомайзере (мокапы уже есть в каталоге) — повышает конверсию.
3. **Дизайн-версии и «ремиксы».** История версий дизайна + кнопка «сделать ремикс» из публичной ссылки чужого дизайна (если автор разрешил) → виральность.
4. **Комплекты/капсулы.** «Закажи всей командой» — один дизайн на N изделий с матрицей размеров (футболка S/M/L/XL) в одном заказе. Готовит почву под B2B.
5. **Программа лояльности / реферал.** Бонусы за повторные заказы и за приведённого друга (промокод-реферал). Цепляется с будущим модулем промокодов админа.
6. **Умные уведомления через WhatsApp** (KZ-рынок живёт в WhatsApp): статус заказа, «ваш дизайн ждёт в корзине», дроп нового принта любимого дизайнера.
7. **Wishlist-алерты.** Принт/товар из избранного попал в дроп/акцию → пуш.

## 2. DESIGNER (`/studio-designer`) — дизайнер

### 2.1 Что РАБОТАЕТ ✅
- Дашборд: заработано / доступно к выводу / ставка / счётчики принтов (pending/approved/rejected) + лента последних продаж (Realtime).
- Загрузка принта: файл в storage, название, теги, согласие с офертой.
- Мои принты: сетка со статусом модерации + `moderation_note` при отклонении, toggle публикации.
- Финансы: баланс (начислено/выплачено/доступно), история начислений (`royalty_earnings` со ставкой и статусом), заявка на выплату (порог 5000₸), история выплат.
- Налоговый статус (физлицо/самозанятый/ИП) + реквизиты выплаты (JSONB).
- Роялти-механика: начисление при оплате через RPC `apply_paid` (идемпотентно, advisory-lock, фиксирует ставку на момент продажи). История ставки `royalty_rate_history` без пересчёта задним числом.
- RLS: дизайнер видит только своё.

### 2.3 ПЛАН — доделать (приоритет)
**P0 (до открытия опоры №2):**
- RPC `refund_order(p_order_id)`: статус → refunded + отрицательные `royalty_earnings` (`reversed`) + пересчёт `designer_balances`. Вызов из webhook возврата.
- Валидация DPI при загрузке (использовать `dpiAtMaxSize` из `shared/config/zones.ts`), блок при DPI < 150.
- Публичная витрина `/designer/[slug]`: bio, аватар, одобренные принты с превью.

**P1:**
- График продаж + топ-принтов на дашборде.
- Карточка принта: продаж, сумма роялти, список изделий, кнопка edit метаданных.
- Фильтры + CSV-экспорт истории начислений (для налоговой).
- Реальная генерация thumbnail.

### 2.4 🔥 Крутые фишки для designer
1. **«Студия дохода» — прогноз и цели.** Прогноз заработка на месяц по тренду, цель «до выплаты осталось N₸», бейджи-ачивки (первая продажа, 10/100 продаж, топ-дизайнер месяца). Геймификация удержания опоры №2.
2. **A/B мокапы и аналитика принта.** Дизайнер видит, на каких изделиях/цветах его принт берут лучше → подсказка «добавь вариант на чёрном худи».
3. **Публичная витрина-портфолио с подпиской.** Покупатель подписывается на дизайнера → пуш о новом принте/дропе. Лицо автора = маркетинг платформы.
4. **Дизайнерские дропы/коллекции.** Дизайнер группирует принты в коллекцию с датой релиза — анонс на лендинге (связка с контент-модулем админа).
5. **Калькулятор «сколько заработаю».** До загрузки: при ставке X% и цене изделия Y роялти = Z. Прозрачность мотивирует.
6. **Авто-проверка качества при загрузке.** Не только DPI: прозрачность фона, цветовой профиль, предупреждение «мелкие детали потеряются на DTF». Снижает брак и нагрузку модерации.
7. **Реферал дизайнеров** (под закрытый старт §7.5): приглашённые по ссылке дизайнеры с трекингом «кто кого привёл».

## 3. OPERATOR (`/studio`) — производство

### 3.1 Что РАБОТАЕТ ✅
- Доска-канбан по этапам queued → printing → quality_check → packing → ready_to_ship (+ on_hold, reprint), **Realtime** (подписка на `orders`).
- Карточка заказа для цеха: спецификация в мм (X/Y/размер/поворот), товар/цвет/размер/SKU/материал/метод, скачивание оригинала в полном разрешении.
- Смена этапа кнопками (state-machine, серверная валидация, атомарный RPC `change_order_status`).
- Брак → reprint и on_hold — с обязательной причиной (modal). Полный лог переходов (актор + причина).
- Отгрузка: shipped с обязательным трек-номером и перевозчиком.
- Склад: остатки по вариантам (подсветка ≤5), фиксация брака (−1, reason=defect).
- Модерация дизайна перед печатью (гейт: нельзя в printing без approved).
- Доступ: только оплаченные заказы (RLS + middleware `studio-role`).

### 3.2 Что ЧАСТИЧНО 🟡 / НЕТ ❌ / 🚨 БАГ
| Функция | Статус | Проблема |
|---|---|---|
| Скрытие цен/себестоимости | 🚨 ❌ | `useStudio.ts`: `order_items(*)` → видны `unit_price`/`unit_cost`; `total` на доске. Нарушение §5.2 |
| Скриншот композиции на карточке | 🟡 | `preview_url` есть в БД, не выводится (только ссылка на оригинал) |
| История движений склада в UI | 🟡 | `listMovements()` есть в composable, но нет экрана |
| Печатный файл (CMYK PDF) | ❌ | `print_file_url` в схеме, логика генерации не реализована |
| Поиск/фильтр по доске | ❌ | Нет фильтра по № заказа/дате/методу |

### 3.3 ПЛАН — доделать (приоритет)
**P0 (security, до запуска):**
- В `useStudio.listQueue()` и `getOrder()` выбирать только производственные поля: `order_items(id, quantity, print_method, variant_id, design_id, ...)`; убрать `total`/`unit_price`/`unit_cost`.
- Лучше — отдельный DB-view `studio_order_items` без финансовых полей (field-level защита, т.к. RLS не фильтрует колонки).

**P1:**
- Вывод `preview_url` (скриншот композиции) на карточке заказа.
- Экран истории движений склада (modal по варианту, `listMovements()` уже готов).

**P2:**
- Поиск/фильтр по доске (№, дата, метод печати).
- Генерация печатного CMYK-PDF при переходе в printing (или явно отложить в Wave 2).

### 3.4 🔥 Крутые фишки для operator
1. **Пакетная печать (batching).** Группировка позиций одного метода/цвета/изделия в «партию печати» → экономия плёнки/времени. Подсказка «5 заказов на DTF на белом — печатать вместе».
2. **Печать наряд-листа / стикеров.** Кнопка «печать ярлыка» с QR заказа, спецификацией и мокапом для цеха и упаковки. QR на упаковке → переход на карточку.
3. **Сканер-режим (мобильный).** Оператор сканирует QR заказа телефоном → сразу карточка + кнопки смены этапа. Цех работает с телефона у станка.
4. **SLA-таймеры и тепловая карта очереди.** Сколько заказ висит на этапе, подсветка просроченных (>X часов), приоритет «горящих». Прозрачные сроки = опора №3.
5. **Чек-лист контроля качества.** На quality_check — чек-лист (цвет совпал / нет дефектов плёнки / позиционирование), фото готового изделия → доказательная база для возвратов (связка с §6.8 админа).
6. **Авто-резерв склада.** При переходе заказа в queued — резервирование заготовки; предупреждение «нет на складе, нужен приход».
7. **Производительность смены.** Сколько единиц/заказов закрыл оператор за день (без финансов) — лёгкая геймификация цеха.

### 4.2 🚨 Критичные баги admin
1. **Прибыль = выручке.** `admin_finance_stats` (`0014_admin_stats.sql:10-11`) не вычитает расходы. Формула должна быть `revenue − cogs − royalty − acquiring_fee − shipping`.
2. **Маржа/себестоимость скрыта в заказах.** `unit_cost` в БД есть, в карточке `/admin/orders/[id]` не показывается — админ может продавать в минус.
3. **Дизайнеры без `designer_balances` пропадают.** Клиентский join в `useFinance.designers()` теряет новых дизайнеров с нулём — нужен LEFT JOIN в RPC.
4. **Аудит дырявый.** Большинство действий (смена цены, снятие товара, модерация, ставка, выплата) не пишутся в `admin_audit_log`.

### 4.3 ПЛАН — доделать (приоритет)
**P0 (до запуска):**
- Переписать `admin_finance_stats`: корректная чистая прибыль из `finance_entries` (revenue/cogs/royalty/acquiring_fee/shipping).
- Показать себестоимость и маржу (`unit_price − unit_cost`) в карточке заказа и в списке.
- Починить список дизайнеров (LEFT JOIN в RPC, не клиентский map).
- Автоматический аудит через триггеры БД на ключевые таблицы (цена, товар, ставка, выплата, модерация).

**P1:**
- Приглашение дизайнеров (генерация инвайтов под закрытый старт §7.5).
- Графики выручки/прибыли/маржи по времени + фильтр периода на дашборде и в финансах.
- Маржинальность по изделию/методу/заказу.
- Промокоды + оптовые пороги (`price_tiers`).
- Редактор зон печати (визуальный, Konva) в мастере товара.

**P2:**
- Доказательная база возвратов (загрузка фото брака + хранение скриншота композиции + оригинала).
- Редактор контент-страниц (FAQ/о нас/политики) + версионирование юр.документов с UI и «применить с даты».
- Справочник поставщиков + история движений склада в UI.
- Налоговый экспорт для бухгалтера (оборот для упрощёнки 4% / доход−расход).
- Справочники методов печати и матрицы совместимости в UI (сейчас хардкод).

### 4.4 🔥 Крутые фишки для admin
1. **Командный центр (живой дашборд).** Realtime-пульс: выручка/прибыль сегодня, заказы по этапам, «что горит» с прямыми ссылками на проблемные заказы. P&L-водопад (выручка → −cogs → −роялти → −эквайринг → −доставка → прибыль).
2. **Юнит-экономика на каждый заказ/SKU.** Маржа по позиции, ROI метода печати, какой принт/изделие приносит больше прибыли (не выручки). Решения на правде.
3. **Конструктор промо-кампаний.** Промокоды (% / фикс / бесплатная доставка), лимиты, сроки, привязка к каналу/дизайнеру; реферальные коды покупателей и дизайнеров в одном модуле.
4. **No-code контент и дропы.** Календарь дропов с авто-публикацией на лендинг, редактор hero/баннеров/страниц с предпросмотром, A/B заголовков.
5. **Модерация-конвейер с ИИ-подсказкой.** В очереди модерации — авто-флаги (возможный копирайт/NSFW/низкое качество/дубликат), массовые действия, шаблоны причин отказа.
6. **Финансовый автопилот.** Авто-формирование `finance_entries` на каждое событие (оплата/возврат/выплата/закуп), сводка для декларации в один клик, экспорт для бухгалтера, алерты «маржа изделия < порога».
7. **Прогнозы и реордеринг склада.** Прогноз расхода заготовок по скорости продаж → «закупить чёрные XL, хватит на 6 дней». Авто-черновик заказа поставщику.
8. **Полный аудит-трейл (forensics).** Кто/что/когда менял, diff before/after, фильтр по сущности и актору — юридическая и операционная защита.
9. **Когорты и LTV.** Повторные покупки, LTV клиента, топ-каналы привлечения, конверсия по воронке (каталог → кастом → корзина → оплата).

## 5. Сквозные улучшения (все роли)
- **Дизайн-система UX:** skeleton-загрузки, осмысленные empty-states с CTA, тосты в фиксированной зоне, breadcrumbs, мобильная адаптация админки/студии.
- **Уведомления как сервис:** единый слой (email + WhatsApp + in-app) с шаблонами по событиям — переиспользуется всеми ролями.
- **Realtime везде, где ждут изменения:** заказы (есть), модерация, выплаты, склад.
- **Типизация RPC-результатов** вместо `unknown as Stats`.
- **Тесты:** покрыть роялти-механику (начисление/reversal/идемпотентность), state-machine статусов, RLS-изоляцию ролей.
