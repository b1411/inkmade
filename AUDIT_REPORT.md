### 🔴 CRITICAL

#### C1. ePay `check-status`: несовместимость путей полей → оплаченный заказ навсегда застревает в `pending` (деньги списаны, заказ не исполнен) — **[новый дефект, критичен на go-live ePay]**
- **Файл/строки:** `server/utils/payment.ts:184-193` (`checkStatus`), потребитель `server/utils/epay-order.ts:20-33`.
- **Воспроизведение:** при `PAYMENT_PROVIDER=epay` покупатель платит → возврат на `/checkout/pay/[id]?returned=1` и/или `post_link`-callback → `syncEpayOrderPayment` → `checkStatus`.
- **Ожидается:** списание → `applyPaid` → `status='paid'`, письмо, исполнение.
- **Фактически:** асимметрия чтения полей ответа Halyk:
  - `statusName` читается с nested-фолбэком: `raw.statusName || raw.transaction?.statusName`;
  - `transactionId` — **только nested**: `raw.transaction?.id`;
  - `amount` / `currency` / `resultCode` — **только top-level**: `raw.amount`, `raw.currency`, `raw.resultCode`.
  `paid = String(raw.resultCode)==='100' && status==='CHARGE'`. Если Halyk кладёт `resultCode`/`amount`/`currency` внутрь `transaction` (как id и statusName), то: `paid` навсегда `false` **или** `amount=NaN` → в `epay-order.ts:24` бросается `ePay amount mismatch` **до** `applyPaid`. Симметрично: если всё лежит top-level, то `transactionId=undefined` → `epay-order.ts:33` бросает `transaction id missing`. При любом раскладе, не совпадающем ровно с этим микс-предположением, заказ не помечается оплаченным.
- **Причина:** непоследовательные предположения о структуре ответа шлюза (nested для одних полей, top-level для других).
- **Риск:** деньги списаны, `applyPaid` не вызван, заказ вечно `pending`, нет исполнения и письма. Повторные «Проверить статус» бросают ту же ошибку. Это ровно тот класс, ради устранения которого делался фикс `Math.round(total)`.
- **Рекомендация:** читать `amount`/`currency`/`resultCode`/`transactionId` с одинаковым фолбэком (`raw.X ?? raw.transaction?.X`); **обязательно** прогнать реальный payload `check-status`/`post_link` в песочнице Halyk перед включением ePay.
- **Уверенность:** асимметрия в коде — **высокая** (проверено лично); реальный триггер зависит от схемы Halyk — **средняя** (нужен sandbox).
- **Примечание о текущем масштабе:** сегодня прод-`paymentProvider` по умолчанию не `epay` (`getPaymentProvider` бросает «not configured»), поэтому живых покупателей это пока не задевает. Но **запуск приёма оплаты = включение ePay = активация этого бага**. Поэтому Critical и блокер запуска платежей.

#### H2. Повтор оплаты ePay переиздаёт инвойс с тем же `invoice_id` → клиент не может доплатить, заказ зависает — **[новый дефект]**
- **Файл/строки:** `server/api/payment/create.post.ts:22, 57-66`; источник `orders.payment_invoice_id` (миграция 0090).
- **Воспроизведение:** клиент инициирует оплату (инвойс создан), бросает страницу банка, возвращается и снова триггерит `/api/payment/create` для того же заказа.
- **Ожидается:** свежая/возобновляемая платёжная сессия.
- **Фактически:** единственный гард — `if (order.paid_at) 409`. Заказ в `pending` заново входит в `createPayment`, который POST-ит `/invoice` с тем же `invoice_id` (уникален на магазин). ePay, вероятно, отклонит дубликат → сырая ошибка на легитимном повторе.

#### M2. `TemplateBrowser` применяет шаблон, уничтожая текущий дизайн и историю undo без подтверждения — **[новый дефект]**
- **Файл/строки:** `app/components/customizer/TemplateBrowser.vue:44` (`applyTemplate`) → `app/composables/useDesign.ts:420-423` (`loadSpec`).
- **Воспроизведение:** собрать дизайн с несколькими слоями → открыть «Шаблоны» → кликнуть любую миниатюру.
- **Ожидается:** merge, либо подтверждение, либо отменяемая замена.

#### M3. `useDesignAutosave`: перекрывающиеся сохранения при латентности > 1с дебаунса → ложный self-conflict, трейлинг-правка не сохраняется до ручного разрешения — **[новый дефект]**
- **Файл/строки:** `app/composables/useDesignAutosave.ts:149-165` (`flush`/`markDirty`), `:107-144` (`saveServer`, оптимистическая блокировка `.eq('revision', …)`).
- **Воспроизведение:** правка → старт сохранения (`status='saving'`); ещё правка; если сеть медленная и `saveServer` дольше ~1с.
- **Ожидается:** трейлинг-правка сохраняется.
- **Фактически:** `markDirty` **всегда** ставит `status='dirty'`, поэтому гард `flush`'а `status==='saving'` не срабатывает. Второй `flush` запускает **второй `saveServer` с тем же базовым `revision`**. Первый успешно апдейтит (revision→R+1); второй с `eq('revision', R)` не находит строку → `.maybeSingle()=null` → устанавливает `conflict`, `status='conflict'`. Пользователь видит диалог конфликта (хотя правки его же, на одном устройстве), а трейлинг-правка не записана до ручного «оставить текущее».
- **Причина:** нет защиты от запуска второго сохранения при незавершённом первом; оба считают `nextRevision` от одной базы.
- **Риск:** периодическая порча UX/потенциальная потеря последней правки на медленной сети в основном конструкторе.
- **Рекомендация:** флаг `dirtyDuringSave` и повторный `flush` после завершения; либо очередь из одного отложенного сохранения; либо не сбрасывать `status` в `dirty` во время `saving`.
- **Уверенность:** средняя (логика проверена; триггер требует латентности > дебаунса).

## 6. Общее заключение о стабильности

**Переработка структурно в неожиданно хорошем состоянии.** Она компилируется, типизируется (0 ошибок TS), проходит 203/203 юнит-теста и собирается в прод. Масштабный демонтаж инфраструктуры (lenis + gsap + vue-konva + ShaderBackdrop + смена шрифтов) выполнен **чисто — с единственным исключением (H1, ZoneEditor)**. В клиентском пути покупателя (лендинг → каталог → товар → конструктор → корзина → checkout → создание заказа) **мёртвых кнопок и битых ссылок не найдено**, SSR/гидратация чисты, гарды авторизации на месте.

**Реальные риски концентрируются в трёх местах, а не «повсюду»:**
1. **Новый платёжный слой ePay** — не проверен против реального API Halyk; **C1** способен ронять оплаченные заказы в вечный `pending`. Это блокер именно запуска приёма оплаты.
2. **Один сломанный админ-холст** (H1) — тривиально чинится, но портит геометрию зон.
3. **Несколько «острых краёв» состояния/данных** — деструктивное применение шаблона (M2), гонка autosave (M3), хардкод-каталог (M1), несовместимость версий спеки (M4).

**Вердикт:** сайт **не «сломан переработкой»** в смысле массовых регрессий. Клиентская витрина и оформление заказа для текущего провайдера исправны. Но **к запуску приёма оплаты не готов** без верификации ePay в песочнице и починки H1/M-находок. Оценка риска: **средний**, с одним критичным латентным блокером платежей.
