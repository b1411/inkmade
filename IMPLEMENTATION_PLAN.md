# INKMADE — План реализации

Рабочий бэклог по паспорту проекта (`INKMADE_project_passport.md`).

Легенда сложности: **S** (≤полдня) · **M** (1–2 дня) · **L** (3–5 дней) · **XL** (неделя+).
🔒 — сквозной инвариант (раздел 10 паспорта), проверяется в каждой релевантной задаче.

**Стек:** Nuxt 4 (SSR) + Nuxt UI v4 + Tailwind v4 + Konva (client-only) → Supabase (Postgres/Auth/Storage/RLS/Realtime/Edge) → Vercel.

**Решения по старту:**
- Платёжный провайдер выбирается позже → заложить абстракцию `PaymentProvider`, чтобы выбор ePay/Kaspi не блокировал Фазу 3.
- Supabase-проект создаём новый.

---

## Три инварианта (раздел 10) — встроить с начала
1. DPI-валидация от **максимального** размера изделия (`products.max_print_mm`).
2. `paid` ставится **только** серверным webhook с проверкой подписи.
3. RLS-политики проверены до релиза.

## Критический путь
F0-1 → F0-7..10 → F1-2..8 → F1-10 → F2-1..3 → F2-11 → F3-1..4 → F3-6..7 → F4-4..5.

## Риски (узлы, где паспорт прямо предупреждает)
1. DPI-валидация (F2-5) — от `products.max_print_mm`, не от текущего размера.
2. Webhook (F3-4) — `paid` только по подписанному webhook.
3. RLS (F0-10/F0-14) — ошибка = утечка чужих данных или повышение роли.
4. px↔мм (F2-2) — без точного пересчёта цех не воспроизведёт макет.
