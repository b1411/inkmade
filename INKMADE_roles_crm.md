# INKMADE — Роли, кабинеты и CRM (документ разработки)

Документ для разработки всех ролей платформы и наполнения функционалом. Дополняет мастер-документ проекта (стек, дизайн-система, схема каталога/печати — там). Здесь: три опоры платформы, четыре роли с детальным функционалом каждой, дизайнерская модель с роялти, полная админ-CRM, расширение схемы БД под роли и финансы.

Стек (из мастер-документа): Nuxt 4 + Nuxt UI v4 + Tailwind v4 + Konva (client-only) на фронте; Supabase (Postgres/Auth/Storage/RLS/Realtime/Edge) на бэке; один платёжный провайдер (ePay/Kaspi) + webhook; деплой Vercel + Supabase. Цвета #7A1F28 / #111111 / #EFE0C1, шрифты граффити-заголовки + Manrope/Inter + моно.

Не входит в объём: блогерская модель (магазины-витрины создателей) — не разрабатывается, не закладывается.

---

## 1. Три опоры платформы

Платформа стоит на трёх несущих опорах. Каждая закрывает свой тип спроса.

1. **Кастомизация** («сделай свою вещь»). Для тех, у кого есть своя идея/файл. Кастомайзер: загрузка, размещение принта в мм, текст. Подтверждённый акцент продукта.
2. **Каталог дизайнеров** (готовые принты с роялти). Для большинства без своей идеи — выбрать красивое и заказать за минуту. Дизайнеры наполняют каталог, платформа контролирует качество модерацией, дизайнер получает процент с каждой продажи своего принта. Несущая опора, не довесок.
3. **Производство и качество** (фундамент под обоими). Собственная печать, контроль качества, предсказуемые сроки. Без него первые две опоры рушатся.

Три источника принтов сходятся в одном каталоге и одном кастомайзере: принты бренда (от платформы) + принты дизайнеров (с роялти) + свой файл покупателя.

**Правило очерёдности возведения:** дизайнерская опора запускается ТОЛЬКО после отлаженного производства. Дизайнер приходит туда, где доказано качество и есть продажи. Позвать художников на сырую платформу = потерять их после первого кривого принта и испортить репутацию в комьюнити. Поэтому: MVP (кастом + принты от платформы + производство) → доказать качество → пригласить 5–10 дизайнеров (закрытый старт опоры) → открыть регистрацию шире.

## 2. Роли и матрица доступа

Четыре роли (`profiles.role`). Блогеров нет.

| Роль | Кабинет | Назначение |
|---|---|---|
| `customer` | `/account` | покупки, свои дизайны, заказы, трекинг |
| `designer` | `/studio-designer` | загрузка принтов, статистика продаж, роялти, выплаты |
| `operator` | `/studio` | производство: очередь, печать, контроль качества, отгрузка |
| `admin` | `/admin` | полная CRM: финансы, дизайнеры, заказы, каталог, юр.документы, аналитика |

Доступ закрыт middleware по роли. На старте operator и admin может быть один человек, но роли разделены в системе.

Иерархия данных по ролям:
- customer видит только своё.
- designer видит только свои принты, свою статистику, свои начисления.
- operator видит производственную сторону всех оплаченных заказов (без финансов и цен-себестоимости).
- admin видит всё.

## 3. Кабинет клиента (`/account`)

Цель: дать покупателю контроль над заказами и сохранёнными дизайнами, минимум барьеров.

**3.1 Разделы**
- **Мои заказы** — список с укрупнённым статусом (оформлен / в производстве / упаковка / в пути / доставлен), сумма, дата. Карточка заказа: позиции, превью композиции, трек-номер со ссылкой на отслеживание, чек.
- **Мои дизайны** — сохранённые в кабинете дизайны (`is_saved`), кнопки «заказать снова», «доработать», «поделиться по ссылке».
- **Избранное** — сохранённые товары и понравившиеся принты из каталога (вторая волна).
- **Адреса** — сохранённые адреса доставки.
- **Профиль** — имя, телефон, email, смена пароля.
- **Подарочные заказы** — если заказ оформлен как подарок: статус, получатель, открытка (подарочный флоу — см. мастер).

**3.2 Поведение**
- Гость собирает дизайн и корзину без логина; логин на checkout перед оплатой; локальные данные переносятся в аккаунт.
- Уведомления о смене статуса (email/SMS/WhatsApp — вторая волна).
- Повторный заказ в один клик из истории.

## 4. Кабинет дизайнера (`/studio-designer`)

Цель: сделать кабинет приятным и мотивирующим — дизайнер должен видеть свой заработок и хотеть загружать ещё. Это инструмент удержания опоры №2.

### 4.1 Дашборд (главный экран)
Сразу при входе — то, ради чего дизайнер пришёл:
- **Заработано всего** и **доступно к выводу** (баланс) — крупно.
- **Продажи за период** — график (день/неделя/месяц): количество и сумма роялти.
- **Топ принтов** — какие его дизайны продаются лучше.
- **Последние продажи** — лента: какой принт, на каком изделии, когда, сколько роялти начислено.
- Быстрые цифры: всего принтов загружено, на модерации, одобрено, отклонено.

### 4.2 Мои принты
- Сетка загруженных принтов со статусом модерации (на проверке / одобрен / отклонён + причина).
- По каждому принту: превью, дата загрузки, число продаж, сумма роялти с него, на каких изделиях заказывали.
- Загрузка нового принта: файл (требования к формату/разрешению/DPI как в каталоге), название, теги, описание, согласие с офертой (права на дизайн — его, ответственность за копирайт на нём).
- Редактирование метаданных, снятие принта с публикации.

### 4.3 Финансы и выплаты
- **Баланс**: начислено, выплачено, доступно к выводу.
- **История начислений**: построчно — продажа, принт, сумма роялти, дата. Прозрачно, чтобы дизайнер доверял расчёту.
- **Заявка на выплату**: при достижении минимального порога. Реквизиты для выплаты.
- **История выплат**: когда, сколько, статус.
- **Ставка роялти**: показывается его персональный процент (задаётся админом индивидуально — см. 7.2). Если процент менялся — видна дата изменения.
- **Налоговый статус**: поле/раздел, где дизайнер указывает свой статус (физлицо / самозанятый / ИП) — влияет на оформление выплат (см. 7.4).

### 4.4 Профиль дизайнера (публичная витрина — опционально, вторая волна)
- Страница автора в каталоге: имя/псевдоним, аватар, его принты. Даёт дизайнеру лицо и продвигает его работы. Это и есть «профиль художника».

### 4.5 Документы
- Оферта/договор с платформой (версия, дата принятия).
- Условия роялти и выплат.
- Налоговое уведомление (его ответственность за свои налоги, если оформлено через оферту).

## 5. Кабинет оператора / производство (`/studio`)

Цель: вести заказ по производственным этапам, видеть «что в типографии, что на отгрузке». Без доступа к финансам, ценам, себестоимости.

### 5.1 Разделы
- **Очередь заказов** — доска по этапам (Realtime, без обновления страницы): queued → printing → quality_check → packing → ready_to_ship → shipped. Видно распределение заказов по этапам.
- **Карточка заказа для цеха**: оригинал принта (скачать в полном разрешении), спецификация нанесения в мм (зона, X/Y, размер, поворот), тип/цвет/размер/материал заготовки, метод печати, количество. Скриншот композиции для наглядности.
- **Смена этапа**: кнопки перевода по автомату. Брак на quality_check → reprint с обязательной причиной. Проблема → on_hold с причиной. Каждый переход пишет лог (актор = оператор).
- **Отгрузка**: ready_to_ship → shipped с вводом трек-номера и перевозчика.
- **Склад заготовок** (просмотр + расход): остатки по вариантам, фиксация брака (defect). Приход заводит админ.

### 5.2 Что оператор НЕ видит
Цены, себестоимость, выручку, данные дизайнеров, финансы. Только производственная информация.

## 6. Админ-кабинет — полная CRM (`/admin`)

Цель: внутри админки есть ВСЁ — от финансов до юридических документов. Это центр управления бизнесом. Ниже все разделы.

### 6.1 Дашборд (сводка бизнеса)
Первый экран — пульс компании:
- **Выручка** за период (день/неделя/месяц/год) — график.
- **Чистая прибыль** = выручка − себестоимость − роялти дизайнерам − комиссии эквайринга − доставка (если за счёт компании). График динамики.
- **Заказы**: количество по статусам, средний чек, конверсия (из внешней аналитики).
- **Топ**: изделия, принты, дизайнеры по продажам.
- **Что требует внимания**: заказы на паузе (on_hold), брак (reprint), дизайны на модерации, низкие остатки, заявки дизайнеров на выплату.

### 6.2 Финансы (полный модуль)
Сердце CRM. Всё про деньги.
- **Доходы**: выручка по периодам, по каналам (кастом / каталог дизайнеров / принты бренда), по изделиям.
- **Расходы**: себестоимость (заготовки + материалы нанесения из material_usage), роялти дизайнерам, комиссии эквайринга, доставка, прочие операционные.
- **Чистая прибыль**: выручка − все расходы, с разбивкой и графиком динамики.
- **Маржинальность**: по каждому изделию, методу, заказу (unit_price − unit_cost).
- **Роялти к выплате**: сколько накоплено по всем дизайнерам, кто готов к выплате.
- **Графики**: выручка/прибыль по времени, структура доходов (по каналам), структура расходов, маржа по категориям.
- **Налоговый учёт**: сводка для декларации (оборот, для упрощёнки 4%; при переходе на ОУR — доход минус расход, НДС). Экспорт данных для бухгалтера.
- **Экспорт**: выгрузка финансовых отчётов (CSV/Excel) за период.

### 6.3 Дизайнеры (управление опорой №2)
Вкладка дизайнеров — открываешь и видишь всё по каждому:
- **Список дизайнеров**: имя, статус, число принтов (загружено/одобрено), всего продаж, заработано, к выплате, персональная ставка роялти.
- **Карточка дизайнера**: вся информация — профиль, его принты с продажами по каждому, проданные дизайны (что, на каком изделии, когда, сумма), количество загруженных, история начислений и выплат, налоговый статус, договор/оферта.
- **Установка персональной ставки роялти** (обсуждается индивидуально — задаётся здесь, фиксируется с датой изменения).
- **Модерация принтов**: очередь загруженных дизайнов на проверку → одобрить/отклонить с причиной (качество, копирайт, запрещёнка).
- **Выплаты**: заявки дизайнеров на вывод, отметка о проведении выплаты, история.
- **Приглашение дизайнеров**: на закрытом старте опоры — генерация приглашений.

### 6.4 Заказы (полная цепочка)
- **Все заказы**: фильтры по статусу (включая производственные этапы), типу (b2c/будущий b2b), каналу, периоду, дизайнеру (чей принт).
- **Полная цепочка отгрузки**: от оплаты до доставки — статус, лог переходов (кто/когда/причина), производственные этапы, трек. Видно где каждый заказ прямо сейчас.
- **Карточка заказа**: позиции, дизайны, спецификации, оплата, себестоимость и маржа позиции, лог статусов, адрес, трек, фискальный чек.
- **Ручное управление**: перевод статуса, on_hold, cancelled, refunded.
- **Возвраты/рекламации**: обработка (см. 6.8).

### 6.5 Каталог (управление товарами)
- **Мастер создания товара** (6 шагов): основное+описание → материалы (автометод по правилу материал→метод→зоны) → варианты цвет×размер (матрица, SKU, остаток) → зоны печати в мм + мокапы → фото галереи → публикация (черновик→опубликовано, предпросмотр).
- Редактирование, снятие с публикации.
- **Принты бренда**: загрузка собственных принтов платформы в каталог (источник принтов №1).
- **Справочники**: методы печати, ткани, матрица совместимости — заводятся один раз, переиспользуются.

### 6.6 Склад
- Остатки по вариантам, приход (закуп заготовок → stock_movements +N), коррекция, история движений, подсветка низких остатков, справочник поставщиков.

### 6.7 Ценообразование
- Управление ценовыми правилами (ступени площади, цвета, подложка, доп. зоны, full-print, стартовый сбор метода).
- Промокоды.
- Оптовые пороги (price_tiers) — на будущее, неактивны на B2C-старте.

### 6.8 Возвраты и рекламации
- Логика ниши: персонализация снимает возврат «по капризу», но НЕ снимает ответственность за брак.
- Обработка заявок: причина, фото брака от клиента, решение (переделка reprint / возврат refunded / отказ со ссылкой на условия).
- Доказательная база: скриншот композиции (что клиент утвердил) + спецификация в мм (что ушло в цех) + оригинал файла (что загрузил). На этом строится разбор «чей дефект».

### 6.9 Юридические документы
- Хранение и версионирование: оферта/ToS, политика конфиденциальности, условия возврата, дисклеймер о цвете, оферта для дизайнеров, согласия пользователей (user_consents — кто, что, когда, версия).
- Редактирование текстов документов (или контент-страниц), публикация на сайт.
- Лог принятых согласий — для юридической защиты при спорах.

### 6.10 Контент и маркетинг
- Управление лендингом (баннеры, hero, тексты блоков).
- Контент-страницы (FAQ, о нас, политики, блог).
- Промо/дропы (анонсы коллекций датами).
- Настройки аналитики/пикселей.

### 6.11 Пользователи и роли
- Список всех пользователей, назначение ролей (customer/designer/operator/admin) — только admin.
- Аудит действий в админке (admin_audit_log): кто менял цену, снимал товар, трогал заказ.

### 6.12 Настройки платформы
- Глобальные параметры: валюта, базовые сроки, контакты, тексты дисклеймеров, минимальный порог выплаты дизайнерам, реквизиты компании, налоговый режим (упрощёнка/ОУР).

## 7. Дизайнерская модель: роялти, выплаты, налоги

### 7.1 Механика (модель фиксированного процента)
Покупатель платит платформе → товар произведён и продан платформой → с каждой продажи изделия с принтом дизайнера ему начисляется роялти = процент от суммы (или от цены принта-компонента, решается единообразно). Деньги копятся на балансе дизайнера → по достижении порога → выплата.

Не наценка дизайнера поверх цены (это маркетплейс-модель, отвергнута), а фиксированный процент с продажи. Контролируемо платформой, цена для покупателя единая.

### 7.2 Персональная ставка
Процент роялти — индивидуальный для каждого дизайнера, обсуждается персонально, задаётся админом в карточке дизайнера. Хранится с историей изменений (дата, старое/новое значение), чтобы начисления за прошлые продажи не пересчитывались задним числом.

### 7.3 Расчёт начисления (момент истины)
Роялти начисляется при подтверждённой оплате заказа (status=paid), не при добавлении в корзину. По каждой позиции с принтом дизайнера: `royalty = sale_base × designer.royalty_pct` (актуальная ставка на момент продажи). Запись в `royalty_earnings`. При возврате/отмене оплаченного заказа — компенсирующая отрицательная запись (роялти аннулируется).

### 7.4 Налоговая развилка (КРИТИЧНО — требует консультанта до запуска опоры)
Массовые выплаты роялти физлицам — налогово тяжёлый узел. Два пути:
- **Платформа — налоговый агент**: удерживает налоги за дизайнера и отчитывается по каждому. Большая бухгалтерская нагрузка при многих физлицах.
- **Дизайнеры — самозанятые/ИП**: сами за себя отчитываются; платформа платит как контрагентам по договору. Через оферту при регистрации ответственность за налоги переносится на дизайнера (распространённая практика POD; законность в KZ подтвердить у юриста).

Рекомендация: путь с переносом ответственности через оферту + поле налогового статуса в кабинете дизайнера. Финальное решение — с налоговым консультантом на реальных числах ДО открытия дизайнерского кабинета. Зафиксировано как блокирующее условие запуска опоры №2.

### 7.5 Стратегия наполнения (решение проблемы пустого каталога)
Курица-яйцо: нет принтов → нет продаж → нет дизайнеров. Решение: не открытый самозалив на старте. Закрытый запуск — платформа сама + 5–10 приглашённых дизайнеров с хорошим персональным процентом наполняют каталог качественными принтами → первые продажи → доказательство «здесь зарабатывают» → открытие регистрации шире. Это решает и наполнение, и контроль качества/мусора.

## 8. Расширение схемы БД под роли и финансы

Дополняет схему из мастер-документа. Здесь — таблицы, специфичные для ролей, дизайнерской опоры и CRM-финансов.

### 8.1 Роль и принадлежность
```sql
-- profiles расширяется ролью designer
profiles (
  id uuid pk fk -> auth.users,
  role text default 'customer',   -- 'customer'|'designer'|'operator'|'admin'
  full_name text, display_name text,   -- псевдоним для витрины дизайнера
  avatar_url text, phone text, email text,
  created_at timestamptz default now()
)
```

### 8.2 Дизайнерский контур
```sql
-- расширение профиля для дизайнера
designer_profiles (
  id uuid pk fk -> auth.users,
  royalty_pct numeric,            -- персональная ставка (задаёт админ)
  tax_status text,                -- 'individual'|'self_employed'|'ip'
  payout_details jsonb,           -- реквизиты для выплаты
  bio text, is_public boolean default false,  -- публичная витрина
  invited_by uuid null, joined_at timestamptz default now(),
  status text default 'active'    -- 'invited'|'active'|'suspended'
)

-- история изменения ставки (чтобы не пересчитывать задним числом)
royalty_rate_history (
  id uuid pk, designer_id uuid fk -> auth.users,
  old_pct numeric, new_pct numeric, changed_by uuid, changed_at timestamptz default now()
)

-- принты дизайнеров (расширение print_library владельцем и модерацией)
-- print_library уже есть в мастере; добавляются поля:
--   owner_id uuid fk -> auth.users (null = принт бренда платформы)
--   moderation_status text default 'pending'  -- 'pending'|'approved'|'rejected'
--   moderation_note text, moderated_by uuid, moderated_at timestamptz

-- начисления роялти (по каждой продаже принта дизайнера)
royalty_earnings (
  id uuid pk,
  designer_id uuid fk -> auth.users,
  order_item_id uuid fk -> order_items,
  print_id uuid fk -> print_library,
  sale_base numeric,              -- база, с которой считается роялти
  rate_pct numeric,               -- ставка на момент продажи (зафиксирована)
  amount numeric,                 -- начислено (+) или аннулировано (−) при возврате
  status text default 'accrued',  -- 'accrued'|'paid'|'reversed'
  created_at timestamptz default now()
)

-- баланс дизайнера (денормализ. сумма; истина — royalty_earnings + payouts)
designer_balances (
  designer_id uuid pk fk -> auth.users,
  total_earned numeric default 0,
  total_paid numeric default 0,
  available numeric default 0,    -- доступно к выводу
  updated_at timestamptz default now()
)

-- выплаты дизайнерам
payouts (
  id uuid pk, designer_id uuid fk -> auth.users,
  amount numeric, method text, details jsonb,
  status text default 'requested',  -- 'requested'|'processing'|'paid'|'rejected'
  requested_at timestamptz default now(), paid_at timestamptz,
  processed_by uuid
)
```

### 8.3 Финансовый учёт компании (CRM)
```sql
-- агрегированные финансовые движения для отчётов и графиков
finance_entries (
  id uuid pk,
  entry_type text,    -- 'revenue'|'cogs'|'royalty'|'acquiring_fee'|'shipping'|'other'
  order_id uuid fk -> orders null,
  amount numeric, currency text default 'KZT',
  note text, created_at timestamptz default now()
)
-- чистая прибыль и графики строятся агрегацией finance_entries по периодам и типам.
-- Источники: orders (revenue), order_items.unit_cost + material_usage (cogs),
-- royalty_earnings (royalty), payments (acquiring_fee), shipping_cost (shipping).

-- аудит действий админа
admin_audit_log (
  id uuid pk, actor_id uuid fk -> auth.users,
  action text, entity text, entity_id uuid, before jsonb, after jsonb,
  created_at timestamptz default now()
)
```

### 8.4 Связь с заказом
`order_items` получает признак источника принта для корректного начисления:
```sql
-- order_items дополняется:
--   print_id uuid fk -> print_library null   (если заказан принт из каталога)
--   print_owner_id uuid null                 (дизайнер-владелец, для роялти; null = бренд/кастом)
```
При оплате заказа: для позиций с `print_owner_id` создаётся `royalty_earnings`, обновляется `designer_balances`, пишется `finance_entries` (royalty как расход).

### 8.5 RLS для новых таблиц
- designer_profiles, royalty_earnings, designer_balances, payouts: дизайнер видит ТОЛЬКО свои строки; admin — все.
- royalty_rate_history: дизайнер видит свою; запись только admin.
- finance_entries, admin_audit_log: только admin.
- print_library: публичное чтение одобренных; дизайнер пишет/редактирует свои (owner_id = auth.uid()), модерацию меняет только admin.

## 9. Порядок разработки ролей

1. **Auth + profiles + 4 роли** + middleware (auth, designer-role, studio-role, admin-role).
2. **Кабинет клиента** — заказы, дизайны, адреса, профиль (нужен для базового флоу покупки).
3. **Админ-CRM ядро** — каталог (мастер товара), заказы с цепочкой, склад. Без этого нечего продавать.
4. **Производство** — /studio, очередь на Realtime, смена статусов, отгрузка.
5. **Финансовый модуль админки** — finance_entries, графики выручки/прибыли/маржи, налоговая сводка.
6. **Дизайнерский контур** (опора №2, после отладки производства):
   - кабинет дизайнера (дашборд, мои принты, загрузка, статистика);
   - модерация принтов в админке;
   - роялти-механика (начисление при оплате, баланс);
   - выплаты (заявки, проведение, история);
   - вкладка дизайнеров в админке (полная информация по каждому);
   - персональные ставки + история.
7. **Юр.документы и согласия** — оферты, ToS, user_consents, версионирование.
8. **Возвраты** — обработка с доказательной базой.
9. **Аудит и настройки** — admin_audit_log, platform_settings.

Налоговую развилку (7.4) решить с консультантом ДО шага 6.

## 10. Сводка по ролям

| Роль | Видит | Не видит | Ключевое |
|---|---|---|---|
| customer | свои заказы/дизайны/адреса | чужое, финансы | покупка, кастом, повтор заказа |
| designer | свои принты/продажи/роялти/баланс | чужое, финансы компании | заработок, загрузка, выплаты |
| operator | производство всех оплаченных | цены, финансы, дизайнеров | очередь, печать, отгрузка |
| admin | ВСЁ | — | CRM: финансы, дизайнеры, заказы, каталог, юр.документы, аналитика |

Роялти: фиксированный процент, персональный per дизайнер, задаётся админом, фиксируется на момент продажи. Дизайнерская опора запускается после производства, по приглашению. Налоговый узел — блокирующий, решается с консультантом до запуска опоры. Блогерская модель в объём не входит.
